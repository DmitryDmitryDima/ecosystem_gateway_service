Сейчас я использую нечто, что можно назвать outbox цепочками  - об этом я подробно напишу в секции java projects сервиса, где на данный момент реализация идеи сильно связана с инфраструктурой spring boot и самим сервисом — в будущем я планирую выделить все в отдельный фреймворк.

По задумке пользователю будут доступен набор полезных возможностей, ориентированных, прежде всего, на обучение и развитие, а также на первоначальное взаимодействие с другими пользователями вне комнат. Доступ к сервису планируется сделать гибким — создать запись можно будет из любого места приложения, в некоторых случаях с возможностью использовать конкретный контекст. Помимо этого, я планирую выделить отдельный storage сервис, позволяющий загружать большие файлы (не путать с индивидуальным дисковым пространством каждого из сервисов). Прежде всего, к загруженному контенту можно будет ссылаться из комнат, используя специальные ссылки — это значительно расширит возможности разработки и ai-генерации. Storage сервис также позволит генерировать карточки на основе загруженных файлов (привет векторным таблицам).

## API Gateway
Является входной точкой в систему.
Используемые технологии -  spring cloud gateway server webflux.
## Auth Service
Работает в паре с api gateway. Источник истины в системе для таких данных, как user uuid, пароль, глобальная роль в системе, никнейм (используем postgresql). С помощью auth сервиса api gateway обогащает запрос, идущий от фронтенда, валидированным security context (см. схему), после чего он попадает во внутренние части системы, тем самым избавляя последние от необходимости заниматься вопросами аутентификации. Аутентификация реализована через JWT (access token + refresh token, способ передачи — http only cookies). 

API Gateway в связке с Auth поддерживают автоматическое определение target UUID и viewer UUID – внутренние сервисы получают security context, где четко разделяется тот, кто смотрит, и тот, чей контент смотрят.

Auth сервис имеет точку соединения с RabbitMQ, являясь источником ивентов, связанных с изменениями хранимых в нем данных. 
Дополнитель auth сервис предоставляет api для resolve операций, необходимых фронтенду для интерпретации данных, пришедших с других сервисов (во избежание денормализации баз данных immutable ключом ко всему в системе является user uuid – через него происходит извлечение контента).


Используемые технологии — Spring Security, Spring Validation, Java JWT, Spring amqp (Rabbit MQ), Spring Data JPA (PostgreSQL), Spring Web.

## Java Projects Service
Внутренний сервис, отвечающий за операции, связанные с java проектами (комнатами). Сразу отмечу, что планирую расширять систему таким образом, что добавление возможности создавать комнату на новом языке будет означать разработку нового микросервиса.
На данный момент находится в фазе активной разработки.
Используемые технологии — Spring Web, Spring Data JPA (PostgreSQL), Spring AMQP (RabbitMQ), Spring Validation, Eclipse JDT Core (анализ кода, пока на стадии изучения), Spring Data Redis (Valkey).

Одной из ключевых особенностей является продвинутая поддержка создания сложных многоступенчатых процессов, с использование outbox паттерна (в будущем планируется внедрение debezium). Операция разделяется на шаги, порядок и детали выполнения которых мы организуем декларативно, с помощью аннотаций. Все операции — полностью event based – исходя из этого, сообщения от них, представленные парой external event context + external event data, попадают в брокер сообщений и далее. Система триггеров позволяет гибко управлять ходом уже существующего процесса (например, остановить процесс), причем управление может быть доверенно как человеку (или нескольким пользователям — поддерживается polling модель), так и ai агенту — api сделан намеренно обезличенным и адаптируемым для различных сценариев.
Ошибки внутри процессов могут быть гибко обработаны через поддержку компенсационного шага, при этом компенсация может быть запущена либо автоматически (Exception), либо вручную (через поле внутри event internal data).
Outbox цепочки поддерживают контроль времени внутри процессов с помощью аннотаций MaxDuration (максимальное время на шаг) и WaitingFor (сколько следующий шаг будет ждать внешнего события, наличие которого является условием перехода к нему)
Подробнее о outbox chains напишу на страничке java projects сервиса. 

## User Service
Данный сервис планируется разрабатывать с целью внедрения социальной составляющей системы, то есть взаимодействий вне комнат (личные сообщения, друзья, блог и т д). Помимо этого, сервис хранит user properties – мутабельные данные пользователя (ссылка на аватар, список друзей  и т д). Планируется доступ к cloud хранилищу, с целью хранения данных чатов, блогов и т.д.

Используемые технологии - Spring Web, Spring Data JPA (PostgreSQL), Spring AMQP (RabbitMQ),  Spring Validation.

## Notification Service
Мост между брокером (RabbitMQ) и websocket слоем (и, соответственно, фронтендом). Предоставляет api, в котором можно получить агрегированные данные об активных соединениях пользователей в комнатах и в других местах системы. Все ивенты, о которых должен знать фронтенд, попадают на этот сервис.

Внешние ивенты разделены по категориям, каждая из которых имеет свою exchange точку. Читая ивенты из соответствующего exchange, notification service расшифровывает event context, на основании которого он производит рутинг. Notification service поддерживает выполнение некоторых действий внутри него (например, при сценариях, когда необходимо вручную убить websocket сессию) — для этого в каждом из контекстов есть опциональные поля — Notification Strategy (позволяет расширить дефолтный рутинг) и Alarm Strategy (стратегия, содержащая список uuid и название действие, которое необходимо выполнить с соединениями, ассоциированные с этими uuid). Обязанность формирования контекста лежит на сервисах, генерирующих ивенты, исходя из этого notification service — лишь исполнитель.

Используемые технологии - Spring AMQP (RabbitMQ), Spring Web, Spring WebSocket.

## AI Service
Jбеспечивает доступ к LLM. На данный момент решение относительно того, стоит ли выделять его в отдельный сервис, не принято.
Главный аргумент «за»  - возможность изолировать долгие и непредсказуемые запросы к LLM, сделав их частью event based системы и outbox цепочек.
Пример — некий процесс между шагом А и Б генерирует ивент запроса к LLM, ответ которой станет условием перехода к шагу Б, одновременно создавая для него все нужные данные (команду). Ивент обрабатывается внутри AI Service, при необходимости получая все необходимые ссылки и информацию от других компонентов системы (тут можно реализовать систему понятных AI сервису команд, по типу readFile()). После получения ответа  от llm в заранее заданном формате, AI сервис генерирует ответный ивент, отправляемый в брокер. Данный ответ, будучи прочитанным на стороне ожидающего сервиса, активирует триггер, которые толкает цепочку дальше к шагу Б.

Главный аргумент «против» - сложность работы с фреймворками (по сути на стороне сервиса, создающего ивент запроса, мне необходимо пользоваться ai инструментами без непосредственного соединения с какой либо llm – это противоречит логике работы с тем же самым Spring AI). 

Используемые технологии — Spring AI.

## Схема работы брокера сообщений
Брокер сообщений предоставляет транспорт для всех событий, возникающих в системе и покидающих пределы конкретного сервиса. Чаще всего они в конечном итоге предназначены для фронтенда, делая его живым и способным обрабатывать и визуализировать «долгие» процессы, взаимодействия и т. д.

Для поддержания такой системы важно добавлять ивенты так, чтобы они имели единый протокол, понятный всем участникам «общения».

Внутри брокера существует набор обменников (exchange). В моем варианте используются fanout разновидности обменников-  то есть событие, будучи опубликованным в обменник, попадет во все очереди (queue), связанные с обменником.

На данный момент существует 4 обменника, каждый из которых характеризует ту или иную категорию ивентов, а также указывает, кто является автором ивента, а точнее процесса, ставшего источником этого ивента:
- users.main_events — сюда публикуются события, связанные с жизненным циклом пользователя и его данными, хранимыми в auth service. Пример — событие смены username. На данным момент окно публикации открыто лишь со стороны auth сервиса, а чтецом с очередью (queue binding) является user service – схема существует на будущее, для демонстрации;
- users.activity_events – сюда публикуются различного рода действия пользователей, их активность по созданию или изменению связанных с ними данных. Существующий на данный момент пример — создание новой комнаты. На данный момент туда публикуется Java Projects Service, в дальнейшем — User Service. Чтецом является notification service, так как подобного рода ивенты, как правило, мгновенно отображаются на фронтенде и, как правило, требуют сложного рутинга.
- users.projects_events – сюда публикуются пользовательские события, относящиеся к действиям внутри комнат («создание файла», «добавление участника», «сообщение из лога запущенного проекта» и т. д.). Исходя из этого, публикуется сюда Java Projects Service, а чтецом является notification service. Живая коллаборация реализуется во многом благодаря этому обменнику.
- system.projects_events -  сюда публикуются системные события, относящиеся к действиям внутри комнат. Пока существует единственный пример — ивент фоновой записи содержимого файла на диск. В дальнейшем этот обменник может сильно пригодится для различного рода действий со стороны ai. Публикуется сюда — Java Projects Service, чтец — notification service.
<img width="978" height="539" alt="image" src="https://github.com/user-attachments/assets/c8d4487c-d8be-4c66-b5cb-89ccf135cf0d" />

> Для организации взаимодействия запрос — ответ внутри системы в планах создать обменники, именованные так же, как сервис, выступающим принимающей стороной. Пример — Java Project отправляет ивент на AI – ai_service.exchange.input, AI отвечает — java_projects.exchange.input. В данном случае вид обменника может быть изменен.
Что касается протокола ивентов. Каждый ивент снабжается заголовкой event_type и строчным значением payload.

Payload – json строка. Она имеет поля:
- message
- type (дублирует заголовок event_type)
- status (примеры — SUCCESS, ERROR, PROCESSING, POLLING)
- data (json строка, объект, расшифровываемые на фронтенде)
- context

Контекст является особенно важным, так как он, будучи специфичным для каждого обменника, расшифровывается на notification сервисе:
Поля контекста, общие для всех ивентов:
- correlation id – как правило ассоциируется с процессом, генерирующим ивенты
- timestamp
- alarm strategy – команда, сопровождаемая списком uuid – в notification service это будет означать, что с каждым находящимся в списке uuid должно быть сделано действие, соответствующее команде;
- notification strategy – два списка uuid, где первый — список тех, в чей приватный канал сообщение должно быть продублировано, второй — то же самое, только канал публичный. Notification strategy расширяет стратегию, дефолтную для каждой из категорий ивентов.

Обе стратегии формируются сервисом — источником. 

Поля, специфичные для конкретной категории, см. в описании notification service и сервисов, генерирующих тот или иной ивент.

## API Gateway и ValidationFilter
В ядре API Gateway лежит файл application.yml. В нем прописан протокол маршрутизации — на схеме вы можете увидеть, какие именно адреса ведут к конкретному сервису.
> Пока что в системе не используется никакого service registry, все адреса — hardcoded. Это будет исправлено на этапе изучения операций деплоя.
Запрос на тот или иной адрес проходит через компонент фильтрации — ValidationFilter. Исключением является /auth/ запросы, адресованные сервису аутентификации. Они не защищены фильтром, так как взаимодействуют с такими эндпоинтами, как /login, /resolve, /register, /logout.

При попадании в фильтр запроса, обращенного к внутренним api сервисам, происходит извлечение из него http cookie с access токеном. В текущей реализации без access токена возможны только get запросы, при этом они все равно валидируются в auth сервисе, так как в дальнейшем для гостей могут понадобится специальные действия.

Помимо этого, извлекается (при наличии) targetUsername, являющийся параметром запроса. 

Если токен присутствует, его нужно валидировать. Для этого обращается на /validate эндпоинт auth сервиса, при этом добавляя извлеченный куки в новый запрос, а также заголовок targetUsername. Так я объединяю авторизацию и resolve функцию в одном запросе.

/validate имеет два сценария ответа:

-  OK – 200. Токен валиден (или же запрашивающий - гость), запрос может быть проброшен в защищенную часть системы. В теле ответа от auth приходит validation response {uuid, username, role, targetUUID}. Запрос обогащается соответствующими заголовками, давая внутренним сервисам доступ к security context, где при необходимости четко разделен viewer и target. Если запрос был гостевой — тело приходит пустым с указание роли GUEST.
-  401 — Unauthorized – токен не валиден, фронтенд получает 401. Как правило, после этого фронтенд инициирует refresh запрос или login.

Таким образом, связка gateway + auth берет на себя всю работу по аутентификации, оставляя внутренним частям системы только авторизационные моменты, возможные благодаря генерации security context.

> В будущем планирую отказаться от перенаправления гостевых get запросов на auth сервер, так как на данный момент это бессмысленно и попросту создает нагрузку.
